## 1. 门禁体系概览

```
G1 (P1 出口)   需求冻结门禁
G2 (P2 出口)   原型就绪门禁
G3 (P3 出口) ★ MVP 发布门禁 ← 可演示闭环
────────────────────────────────────── MVP 分界线
G4 (P4 出口)   体验完整体门禁
G5 (P5 出口)   测试通过门禁
G6 (P6 出口)   实验完成门禁
G7 (P7 出口) ★ 最终交付门禁 ← 全量发布
```

### 门禁执行规则

- 每阶段结束时由 SE 牵头执行门禁检查
- **阻塞项** (Blocker): 必须全部通过，否则不可进入下一阶段
- **建议项** (Advisory): 尽量完成，可附带缓解计划进入下一阶段
- 门禁结果记录在 `开发进度打卡表.md` 对应 checkpoint 中

## 4. 通用验收标准 (跨阶段)

### 4.1 代码质量标准

| 维度 | MVP 标准 (P3) | 全量标准 (P7) |
|------|:--:|:--:|
| 单元测试覆盖率 | ≥ 80% (信号处理核心) | ≥ 90% |
| 压力测试 | 600 帧零崩溃 | 36000 帧 (1h) 零崩溃 |
| 代码审查 | SE 自审 | 交叉审查 (至少 1 名队友) |
| 文档注释 | 公共 API docstring | 全部 public 函数 |
| Git 提交 | 原子化提交, 有意义的消息 | 同 MVP |
| 无 console.error/warning | 关键路径 | 全量 |

### 4.2 设计审查标准

| 维度 | MVP 标准 (P3) | 全量标准 (P7) |
|------|:--:|:--:|
| 视觉一致性 | 风暴场景通过 | 4 场景全部通过 |
| 色板像素限制 | ≤ 32 色/场景 | ≤ 16 色/场景 |
| 像素单位 | PPU=32 | PPU=32 |
| 呼吸引导圈流畅度 | > 20 FPS 视觉 | > 30 FPS 视觉 |
| 监控台可读性 | 数字可见 (2m 距离) | 图表 + 数字 (3m 距离) |

### 4.3 完成定义 (DoD)

一个工作包标记为「完成」需满足全部条件：

1. 代码/资产已提交到 Git 对应分支
2. 关联的单元测试通过 (如适用)
3. 通过本阶段门禁对应的检查项
4. 队友确认接收 (如有交接)
5. 相关文档/注释已更新

## 5. 术语合规审计清单

### 5.1 禁用语词 (红色)

以下词语及其中文变体禁止出现在用户界面、提示词、文档正文中：

| 禁止词 | 建议替代 |
|--------|----------|
| 判断 / diagnose | 评估 / assess |
| 干预 / treat / therapy | 训练 / training |
| 病理问题 / disease / illness | — (不使用此概念) |
| 参与者 / patient | 用户 / user / participant |
| 消费级设备 / medical device | 消费级设备 / consumer device |
| 正式评估 / clinical | 实验 / experimental |
| 症状 / symptom | 体验 / experience |
| 康复 / recover (medical) | 恢复平静 / regain calm |
| 处方 / prescribe | 建议 / recommend |

### 5.2 推荐语词 (绿色)

| 使用场景 | 推荐用语 |
|----------|----------|
| 系统定位 | 情绪调节教育交互系统 |
| 数据解释 | 交互状态估计 (Interaction State Estimation) |
| 目标用户 | 大学生 / 参与者 / 用户 |
| 效果描述 | 帮助恢复平静、提升觉察 |
| 评分含义 | 反映当前呼吸-身体状态的变化趋势 |

### 5.3 审计流程

1. **P3 MVP 发布前**: 人工审查所有用户可见界面文字
2. **P5 测试前**: `grep` 脚本扫描全部 `.md` `.py` `.cs` `.toe` 文本
3. **P7 交付前**: 终审，包含 PPT + 论文 + 报告
4. **工具**: `rg -i '判断|病理问题|干预|参与者|专用|正式评估|症状|康复|处方' --type-add 'toe:*.toe'`

## 6. 性能基准与基线

### 6.1 延迟预算

```
总延迟 = Python 处理 + UDP 传输 + TD/Unity 渲染
目标: < 200ms (端到端)

Python 信号处理:    < 50ms  (buffer + filter + score)
UDP 传输:          < 5ms   (localhost loopback)
TD 解析 + 渲染:     < 100ms (JSON parse + CHOP update + draw)
Unity 解析 + 渲染:  < 100ms (JSON parse + particle update + draw)
```

### 6.2 吞吐量要求

| 指标 | 目标 | 测量方法 |
|------|:--:|----------|
| UDP 帧率 | 10Hz ± 1% | Python time tick 日志 |
| TD 帧率 | ≥ 30 FPS | TD perform 窗口 FPS |
| Unity 帧率 | ≥ 30 FPS | Unity Stats 面板 |
| CSV 写入 | 不阻塞主循环 | 异步 writer thread |
| 内存使用 | < 500MB | Task Manager |

### 6.3 压力测试基线

| 测试 | 指标 | 通过线 |
|------|------|:--:|
| 短时运行 | 600 帧 (1min) | 零崩溃, 零 NaN |
| 中时运行 | 6000 帧 (10min) | 零崩溃, 丢帧 < 1% |
| 长时运行 | 36000 帧 (1h) | 零崩溃 |
| 多天气切换 | 每 30s 切换天气, 连续 5 次 | 切换不崩不卡 |

## 7. 缺陷严重等级与处理策略

| 等级 | 定义 | 示例 | MVP 策略 | 全量策略 |
|:--:|------|------|:--:|:--:|
| **Blocker** | 核心流程无法运行 | main.py 崩溃, UDP 不发, CSV 不写 | 必须修复 | 必须修复 |
| **Critical** | 功能缺失导致无法体验 | 引导圈不缩放, 风暴粒子不变化 | 必须修复 | 必须修复 |
| **Major** | 功能可用但体验显著受损 | 数值延迟 > 500ms, 颜色明显不符 | 必须修复 | 必须修复 |
| **Minor** | 小问题但可接受 | 字体大小略偏, 粒子偶尔闪烁 | 记录不修 | 尽量修复 |
| **Cosmetic** | 审美瑕疵 | 间距微调, 颜色微调 | 忽略 | 记录不修 |

### 缺陷处理时限

| 等级 | MVP 阶段 | 全量阶段 |
|:--:|:--:|:--:|
| Blocker | 4h 内 | 2h 内 |
| Critical | 24h 内 | 8h 内 |
| Major | 48h 内 | 24h 内 |
| Minor | 记录不修 | 下一迭代 |
